# AGENTS.md

## Purpose
This is the **operational contract** for human and AI agents working in this repo.

**Non-optional rule:** if you change architecture, runtime behavior, scripts, env vars, ports/auth, folder structure, or developer workflow, you must update this file in the same change.

---

## Monorepo shape
- `app/` — Tauri desktop app (React + Vite + TypeScript frontend, Rust backend shell)
- `server/` — Bun + Elysia HTTP server, compiled to a sidecar binary for Tauri

High-level flow:

```mermaid
graph LR
  UI[React UI in app/src] -->|HTTP| API[Bun/Elysia server]
  Tauri[Rust Tauri host] -->|spawns sidecar| API
  API --> Health[/health, /health/ready]
```

To swap the local-dev logo, replace files inside `app/src-tauri/icons/local/` (keep filenames identical so `tauri.localdev.json` doesn't need editing) and re-run `bun run local:dev` so Cargo re-embeds the Windows resource.

## Important runtime behavior

### Server readiness + health
- `server/src/readiness.ts` gates app routes until server marked ready.
- `GET /health` always returns `{ status: "ok", version }`.
- `GET /health/ready` returns ready status or 503 starting.

### CORS + auth in server
- CORS allows `http://localhost*`, `tauri://localhost`, `http(s)://tauri.localhost`.
- If `SERVER_PASSWORD` is set, Basic auth is required (`app:<password>`).

### Frontend connection monitor
- `app/src/features/server/*` polls `${SERVER_BASE_URL}/health` every 3s, where `SERVER_BASE_URL` (in `app/src/lib/server-url.ts`) resolves to `http://127.0.0.1:4728` in Vite dev (`import.meta.env.DEV`) or `http://127.0.0.1:4727` in built bundles. `VITE_API_URL` overrides both.
- `waitForServerConnection()` is a global gate used by `app/src/lib/api.ts` before HTTP requests.

### Global settings
- Global app settings are stored at `~/.agnt/settings.json`.
- Supported settings categories are `hotkeys` and `toolPermissions`.
- Legacy/unknown keys are stripped and rewritten on load so removed categories do not persist.

### Agentic modes
- Conversations support two agentic modes: `agent` (default, full editing) and `plan` (read-only + plan creation). Stored as conversation-level state key `agenticMode` via the existing history/state system.
- **Agent mode**: all tools available except `write_plan`. This is the default implementation mode.
- **Plan mode**: restricted tool set — `read_file`, `glob`, `grep`, `use_skill`, `question`, `todo_write`, `shell`, `await_shell`, `web_search`, `web_fetch`, `write_plan`. No file-mutation tools (`write`, `str_replace`, `apply_patch`, `image_gen`). System prompt includes plan-mode instructions telling the model to research and create a plan.
- Mode is resolved in `resolveConversationModelSettings()` in `conversation.stream.ts` and passed to `buildConversationTools()` via `getAgenticMode` on the permission context. Tool filtering happens in `with-permission.ts` — `PLAN_MODE_TOOLS` set controls which tools are available in plan mode.
- Frontend: `AgenticModeSelector` in the chat input bar (next to `PermissionModeSelector`). `useAgenticMode` hook reads/writes the `agenticMode` state key. Hotkey: `Ctrl+Shift+M`.
- Plan files are stored at `~/.agnt/plans/plan-<uuid>.md`. The `plans` SQLite table links each plan to a conversation (one plan per conversation, cascade delete). Plan service at `server/src/modules/conversations/plans/plans.service.ts`.
- `write_plan` tool (ungated) creates/updates the plan. Emits `plan-updated` SSE event. Frontend auto-opens the Plan tab in the right sidebar on this event.
- Plan panel in the right sidebar (`app/src/features/plans/PlanPanel.tsx`) renders plan markdown + todos list + "Build" button.
- Build button: calls `POST /workspaces/:id/conversations/:conversationId/plan/build` which injects plan todos as conversation todos (all `pending`), sets `agenticMode` to `agent`, then the frontend sends a "Build according to the plan" message to start implementation.
- Plan endpoints: `GET/DELETE /workspaces/:id/conversations/:conversationId/plan`, `POST .../plan/build`.

### Subagents (`task` tool)
- The `task` tool spawns hidden child conversations ("subagents") as rows in the same `conversations` table, marked with `parent_conversation_id`, `subagent_type`, `subagent_name`, and `hidden=1` (migration in `server/src/lib/db.ts`; indexes `idx_conversations_parent` and `idx_conversations_visible`). The sidebar listing filters on `hidden=0 AND parent_conversation_id IS NULL`, so subagents never appear there.
- Subagent types and their tool scopes (one source of truth at `server/src/modules/conversations/subagents/subagent-types.ts`): `generalPurpose` (all tools except `task` + `write_plan`), `explore` (read-only research: `read_file`, `glob`, `grep`, `web_search`, `web_fetch`, `use_skill`, `todo_write`, `question`), `shell` (`shell`, `await_shell`, `read_file`, `glob`, `grep`, `use_skill`, `todo_write`, `question`), `docs` (documentation specialist: `read_file`, `glob`, `grep`, `use_skill`, `web_fetch`, `question`), `best-of-n-runner` (same as `generalPurpose` for now; a TODO there marks it as the future git-worktree-isolation slot). The `task` tool itself is always filtered out for subagents, enforcing single-level nesting (a subagent cannot spawn further subagents).
- Each subagent receives a random curated firstname from `server/src/modules/conversations/subagents/names.ts` (`pickSubagentName()`), stored on the row and exposed via breadcrumbs + the `TaskBlock` card.
- Lifecycle (`server/src/modules/conversations/subagents/subagent-runner.ts` + `subagent-registry.ts`): `task.execute()` calls `runSubagent({ workspaceId, parentConversationId, subagentType, prompt, model?, reasoningEffort?, parentAbortSignal })` which (1) picks a firstname, (2) inserts a hidden conversation + the initial user message via `createSubagentConversation`, (3) chains a child `AbortController` to the parent's signal so parent-stop cascades, (4) calls `runSubagentStream` in `conversation.stream.ts` which runs a normal `streamText` loop with `subagentType`-filtered tools and `forceAgenticMode: "agent"` (subagents always run in agent mode — plan-mode parents can *spawn* subagents but the spawned subagent itself always has write tools available), and (5) awaits the final assistant text to return it through `task`'s `toModelOutput` narrowing (so the parent's context stays lean — only the final text re-enters the parent's prompt).
- Event broadcasting (`server/src/modules/conversations/conversation-events.ts`): a per-conversation-id pub/sub (`EventEmitter`) that `conversation.stream.ts` tees its `controller.enqueue(...)` calls through. The parent's primary `/stream` or `/reply` endpoint wraps its controller with `wrapControllerWithBroadcast` so existing behavior is unchanged; additional observers (like the subagent's `/events` viewer) can subscribe via `subscribeToConversationSse(conversationId)` and receive the same SSE bytes live. `publishConversationSseLine` is also used to fan out the subagent lifecycle events (`subagent-started` / `subagent-finished`) to the parent's listeners so a parent viewer sees subagent spawn/finish without polling.
- Cascade-stop: `stopGeneration` on a parent aborts that parent's HTTP request as before; server-side the abort path in `runStreamTextIntoController` now calls `abortSubagentsForParent(conversationId)` which iterates `subagent-registry.ts`'s in-memory `Map<subagentId, { parentId, abortController, ... }>` and aborts every descendant's `AbortController`. Deleting a parent cascades the same way plus deletes the rows.
- Model selection: subagents default to `gpt-5.4-mini` with `high` reasoning effort. `resolveConversationModelSettings` in `conversation.stream.ts` learns a new precedence for rows with `parent_conversation_id`: (1) `task` tool-call overrides (`model` arg) > (2) parent conversation's `subagentModel` / `subagentReasoningEffort` state keys (written by the chat `ModelSelector`'s new "Subagent model" section, scoped per-conversation or workspace-wide via the existing `mergeScopeState` / `getEffectiveConversationState` path) > (3) hard-coded `DEFAULT_SUBAGENT_MODEL` / `DEFAULT_SUBAGENT_REASONING_EFFORT`. The UI defaults (`DEFAULT_SUBAGENT_MODEL_ID`, `DEFAULT_SUBAGENT_REASONING_EFFORT`) in `app/src/features/models/use-model-selection.ts` are kept in sync with the server.
- Routes (`server/src/modules/conversations/conversations.routes.ts`):
  - `GET /workspaces/:id/conversations/:parentId/subagents` → returns `{ subagents: Conversation[] }` via `listSubagents`.
  - `GET /workspaces/:id/conversations/:conversationId/events` → read-only SSE observer that subscribes to the broadcaster and forwards every SSE line for that conversation id. Used by the subagent page to render a live view without holding its own stream controller.
- SSE events added on top of the existing protocol: `subagent-started` (`{ parent_conversation_id, messageId, subagent: { id, parentConversationId, subagentType, subagentName, title, startedAt } }`) and `subagent-finished` (`{ parent_conversation_id, messageId, subagent_id, outcome: "success"|"error"|"aborted", final_text, error, ended_at }`). Both are emitted on the parent's stream (so the parent UI updates in real time) and on the subagent's own stream (for the observer endpoint).
- Frontend (`app/src/features/conversations/`): `Conversation` type extended with `parent_conversation_id`/`subagent_type`/`subagent_name`/`hidden`; `ToolInvocation` gained `subagent_id` linked live from `subagent-started` + rehydrated from `output.subagentId` on history fetch. New `subagentsByParentId: Record<string, Conversation[]>` store slice + `loadSubagents(workspaceId, parentId)` API helper. `observeConversation(workspaceId, conversationId)` opens the `/events` SSE and reuses the normal SSE reducer pipeline for read-only updates. `handleConversationSseEvent` handles the two new event types plus links `subagent_id` onto the first pending `task` tool invocation on `subagent-started` so the `TaskBlock` card becomes clickable immediately.
- Frontend UI: `ToolCallCard.tsx` renders a `TaskBlock` for `task` invocations (RobotIcon header `"<Name> · <Type> · running|done"`, subtitle = `description`, body = live assistant-text snippet while running, final text once done; wrapped in a TanStack `Link` to `/conversations/<subagentId>` once the id is known). On subagent routes (`app/src/routes/conversations/$conversationId.tsx`) the normal title `Popover` is replaced with a breadcrumb row (`<parent title (Link)> > <subagent-name> [type-pill]`); the route starts `observeConversation` for that subagent and `loadConversation` for the parent so the parent title resolves for the link. The `ModelSelector` popover gained a "Subagent model" section with its own model + reasoning-effort submenus that write to `subagentModel` / `subagentReasoningEffort` through the same scope-aware state path as the primary model.

### Conversation storage (SQLite)
- Each workspace has a SQLite database at `~/.agnt/workspaces/<workspaceId>/conversations.db`.
- Tables: `conversations` (id, title, created_at, updated_at, plus `archived_at TEXT` — NULL = active, ISO timestamp = soft-archived; the sidebar's main list filters `archived_at IS NULL`, the archived section filters `archived_at IS NOT NULL`), `messages` (id, conversation_id, role, content, persisted reasoning fields `reasoning_content`/`reasoning_started_at`/`reasoning_ended_at`, token columns `input_tokens`/`output_tokens`/`reasoning_tokens`/`total_tokens`, `compacted` flag, `summary_of_until` for compacted summary rows, `model_id` for the resolved model that produced the assistant message, `generation_duration_ms` for the active wall-clock duration spent generating the message excluding periods blocked on permission/question prompts), `attachments` (adds `estimated_tokens`), `state_entries` (latest workspace/conversation key-value state), `history_entries` (append-only workspace/conversation state history), `tool_invocations` (id, message_id, tool_name, input_json, output_json, error, status, created_at) linked to assistant messages with cascade delete, `todos` (id, conversation_id, content, status `pending`/`in_progress`/`completed`/`cancelled`, sort_index, created_at, updated_at) cascade-on-conversation-delete, and `plans` (id, conversation_id UNIQUE, file_path, title, todos_json, created_at, updated_at) cascade-on-conversation-delete.
- `server/src/lib/db.ts` manages per-workspace DB instances with caching and auto-migration.
- Conversations are created lazily on first user message.
- Archive vs delete: the conversation row's hover button on the sidebar archives (soft, reversible) instead of deleting. Archive is non-destructive — it only flips `archived_at` to a timestamp and does NOT abort streams, clear pending permissions/questions, or drop in-memory caches. Endpoints: `POST /workspaces/:id/conversations/:conversationId/archive` (sets `archived_at = now`), `POST .../unarchive` (sets `archived_at = NULL`), `GET /workspaces/:id/conversations/archived` (lists rows with `archived_at IS NOT NULL`, ordered by `archived_at DESC`). Hard delete (`DELETE /workspaces/:id/conversations/:conversationId`) still exists and is invoked from the per-row trash icon inside the Archived popover, gated behind a confirm modal — it aborts the stream + clears permission/question/todo state + cascade-deletes the messages/tool_invocations/reasoning/todos/plans/attachments rows. Frontend: `archivedByWorkspace` slice + `archiveConversation`/`unarchiveConversation`/`loadArchivedConversations` actions in `app/src/features/conversations/conversation-store.ts`. The Archived view is a per-workspace floating `Popover` (no persistent UI footprint): the workspace row's existing `⋯` `Menu` gained a `View archived` `Menu.Item` (with `ArchiveIcon`) that flips a per-row `archivedOpen` state to open a `Popover` anchored to the row via a zero-size hidden `<span>` `PopoverTrigger` next to the menu. Each `WorkspaceRow` (extracted from `WorkspaceSidebarList` so each can own its own popover state) renders the popover (`side="right"`, `align="start"`, `sideOffset={8}`, `w-72`) containing a header (`Archived in <workspace> · N`) and a `WorkspaceArchivedList` of the chosen workspace's archived rows with two hover-revealed icons each (`ArrowCounterClockwiseIcon` Restore + `TrashIcon` permanent-delete that opens `ConfirmDeleteDialog`); clicking a row navigates + closes the popover. Loading is lazy: `loadArchivedConversations` is called on `Menu.Item` click and again from `WorkspaceArchivedList` on mount. Both archive and unarchive update the store optimistically and revert on API failure.
- Active conversation streams can be cancelled from the frontend stop button; the frontend aborts the HTTP request, the server forwards `request.signal` into AI SDK `streamText`, and partial assistant text is persisted while empty aborted placeholders are removed.
- Assistant reasoning text and reasoning start/end timestamps are persisted on the message row and returned by conversation fetches so completed/aborted thinking survives app restart and reload.
- Workspace-level and conversation-level metadata/history are exposed under `/workspaces/:id/state|history` and `/workspaces/:id/conversations/:conversationId/state|history`, with `state/effective` providing workspace defaults merged with conversation overrides.
- `GET /models` returns the frontend model catalog used by the chat input selector.
- Conversation streaming resolves model settings from effective state keys: `activeModel`/`model`, `reasoningEffort`/`effort`, and `fastMode`; fast mode maps to OpenAI priority processing when the selected model supports it.
- Conversation system prompts are now composed in `server/src/modules/conversations/conversation.prompt.ts` and include the base assistant instructions, workspace path metadata, and any injected repo instructions resolved from workspace files.
- Repo instruction discovery lives in `server/src/modules/conversations/repo-instructions.ts`. It reads `AGENTS.md`, `CLAUDE.md`, `.agents/AGENTS.md`, and `.claude/CLAUDE.md` from the active workspace root, applies truncation/budget limits, caches by workspace path + file mtimes, and exposes inspection data at `GET /workspaces/:id/repo-instructions`.

### Skills
- Skills are reusable playbooks stored on disk as a directory containing a `SKILL.md` with frontmatter `name:` and `description:`, plus any bundled resource files.
- Discovery roots (later entries override earlier ones on name collision; project always wins over user):
  - User:
    1. `~/.agnt/skills/<name>/SKILL.md`
    2. `~/.agents/skills/<name>/SKILL.md`
    3. `~/.claude/skills/<name>/SKILL.md`
  - Project (inside the active workspace):
    1. `<workspace>/.agnt/skills/<name>/SKILL.md`
    2. `<workspace>/.agents/skills/<name>/SKILL.md`
    3. `<workspace>/.claude/skills/<name>/SKILL.md`
- `server/src/modules/skills/skills.service.ts` walks these roots each stream. Found skills are (a) listed by name/description in the system prompt under `<available_skills>` via `buildAvailableSkillsBlock`, and (b) exposed to the `use_skill` tool through `buildConversationTools({ getSkills })`.
- `GET /workspaces/:id/skills` returns `{ workspacePath, userSkillsDirs, projectSkillsDirs, skills: [{ name, description, directory, source }], warnings }` for inspection.
- `use_skill(name)` tool loads a single skill's full `SKILL.md` body plus a listing of bundled files. Success returns `{ ok: true, name, description, source, directory, content, files }`. Not-found returns `{ ok: false, error, requested, available }` (no throw, so the agent can recover).
- The agent is instructed to call `use_skill` the moment a task matches a listed skill, and never to fabricate skill contents.

### Slash commands (chat input `/`)
- The chat input accepts slash commands **anywhere** in the message — typing `/` at start-of-doc or after whitespace (`allowedPrefixes: [" ", "\n", "\t"]`, `startOfLine: false` in `SlashSuggestion.tsx`) opens a popover listing built-in mode commands + every discovered skill, with arrow-key/Enter selection. Mid-word `/` (e.g. `path/to/file`, `1/2`, `http://`) does NOT trigger the popover. Picked skill commands stay inline as plain text wrapped in a Tiptap `Mark` named `slashCommand` (class `slash-command-token`) styled neon-yellow via `app/src/index.css`; the text is regular editable text (not a node/chip), serializes plainly into the user's message, and the slash mark auto-disappears if the user edits the run away from a slash token.
- Two command kinds:
  - **Mode commands** (`/agent`, `/plan`, `/ask`, `/bypass`) — fire **immediately on popover-select** (no Enter required). The Suggestion plugin's `command` handler delegates to the host via the `onSelectCommand` extension option, which `ChatInput`'s `handleSlashSelect` wires to `useAgenticMode().setAgenticMode(...)` / `usePermissionMode().setPermissionMode(...)` and returns `"consumed"` so the plugin just deletes the typed `/<query>` range without inserting a marked token. Pure UI toggle — never sends a message, never leaves anything in the editor. A regex fallback in `handleSend` (`extractLeadingSlashCommand`) still catches mode commands typed-and-Entered without using the popover (e.g. `/agent ` then Enter), preserving the same toggle-and-bail behavior. Hard-coded list lives in `BUILT_IN_SLASH_COMMANDS` in `app/src/features/slash-commands/slash-api.ts`.
  - **Skill commands** (`/<skill-name>`) — popover-select inserts the marked `/<name>` token at the cursor position (anywhere in the message). On send, the editor's `serialize()` walks the doc, extracts every `slashCommand`-marked text into `slashCommandNames`, and **excludes** the marked text from the message body (one adjacent whitespace is also suppressed so we don't leave double-spaces where the token used to be). Skill names resolved against the cached command list are forwarded as `useSkillNames: string[]` on the existing send paths. The server resolves each name against the workspace's discovered skills, injects the full `SKILL.md` body into the turn-only system prompt under an `<active_skills>` block (NOT in the cached `instructions` field — see prompt-caching note below), and additionally inserts a synthetic `use_skill` `tool_invocations` row + emits `tool-call`/`tool-result` SSE pair so the chat UI's existing `ToolCallCard` shows the auto-loaded skill exactly as if the model had called `use_skill` itself.
- Frontend module: `app/src/features/slash-commands/` — `slash-types.ts` (shared `SlashCommand` type with `kind: "skill" | "mode"`, `mode?: { kind: "agentic"|"permission"; value }`, `source?: "user"|"project"`), `slash-api.ts` (built-in mode commands constant + `fetchWorkspaceSkillCommands(workspaceId)` wrapping `GET /workspaces/:id/skills`), `slash-store.ts` (in-memory cache + `loadSlashCommands(workspaceId)` / `prefetchSlashCommands(workspaceId)` / `readCachedSlashCommands(workspaceId)`; built-in commands win on name collision so a skill literally named `plan` can never shadow `/plan`), `parse-slash.ts` (`extractLeadingSlashCommand(text): { command: string | null; rest: string }`), `index.ts` (barrel).
- New Tiptap pieces under `app/src/components/chat/editor/`:
  - `SlashCommandMark.ts` — Tiptap **Mark** (not a Node) named `slashCommand`, `inclusive: false` and `spanning: false` so the mark can't grow into adjacent typing. Class `slash-command-token` is recolored neon-yellow (`#faff00`, `font-weight: 500`, subtle text-shadow) in `app/src/index.css`.
  - `SlashSuggestion.tsx` — Tiptap `Suggestion` plugin configured with `char: "/"`, `startOfLine: true`, separate `PluginKey<SlashSuggestionState>("slash-suggestion")`, and the same DOM positioning logic + `setPopupOpen` counter pattern as `MentionSuggestion`. Insertion replaces the typed `/<query>` with `/<name>` wrapped in `slashCommand` mark + a trailing space. Exposes `isSlashPopupOpen()` and `isSlashPopupActive(view.state)` for external Enter-submit guards.
  - `SlashList.tsx` — popover list (sections "Modes" + "Skills" with `LightbulbFilamentIcon` + `SparkleIcon`), filter+sort+arrow-key/Enter navigation, fetches via `loadSlashCommands(workspaceId)`.
- `MentionEditor.tsx` registers `SlashCommandMark` + the slash suggestion extension, also calls `prefetchSlashCommands` next to the existing mention prefetch, and its `handleKeyDown` now gates Enter on `isSlashPopupOpen()` / `isSlashPopupActive(view.state)` in addition to the mention popover guards. New `setPlainText(text)` method on `MentionEditorHandle` lets `ChatInput` programmatically replace the editor with plain text after intercepting a mode command.
- `ChatInput.tsx` `onSend` signature gained an optional 4th arg `useSkillNames?: string[]`. `handleSend` now: (1) bails if either popover is open or active; (2) calls `extractLeadingSlashCommand(serialized.text)`; (3) if the parsed command is a mode command (resolved against `readCachedSlashCommands(workspaceId)`), calls `setAgenticMode` / `setPermissionMode`, replaces the editor text with `rest` via `setPlainText`/clear, and returns early; (4) if it's a skill command, strips the command from the outgoing text and passes `[skillName]` as the 4th arg. `canSend` was widened so a slash-only message (no other content, no attachments, no mentions) is still sendable for mode commands.
- `useSkillNames` is threaded through every send path: home `/` route (`createConversation`), conversation route (`sendMessage`), `useConversationStore.createConversation` / `sendMessage`, and `conversationApi.createConversation` / `streamMessage` (gets serialized into the request body of `POST /workspaces/:id/conversations` and `POST .../conversations/:conversationId/stream`).
- Backend wiring (`server/src/modules/conversations/conversations.routes.ts`): both the create-conversation and `/stream` routes parse `useSkillNames?: string[]` from the body via the shared `sanitizeSkillNames` helper (`MAX_SKILL_NAMES_PER_TURN = 4`, `SKILL_NAME_PATTERN = /^[a-zA-Z][\w-]*$/`, dedupes + trims). Forwarded into `createConversation(...)` and `streamConversationReply(...)`. The `/reply` endpoint (used by the home-screen create-then-reply flow) does NOT accept a body field directly — instead it picks the names back up from the new `messages.use_skill_names TEXT` column on the most recent user row (see DB note below) via `readSkillNamesFromLatestUserMessage` inside `streamReplyToLastMessage`.
- DB column: `messages.use_skill_names TEXT` (JSON-encoded `string[]`) added in `server/src/lib/db.ts` (both `SCHEMA` and `runMigrations` via `addColumnIfMissing`). Persisted on the user row in `createConversation` (so the create-then-reply home path can pick the names up later) and on the appended user row in `streamConversationReply` (so the same data is recoverable for replay/audit even though the stream layer also receives it as a function arg).
- Server stream injection (`server/src/modules/conversations/conversation.stream.ts`): inside `runStreamTextIntoController`, after `prompt.todosBlock` is appended as a trailing system message, requested skill names are resolved against the discovered skills via `findSkill(name, skills)`, then `buildActiveSkillsBlock(skills)` (added next to `buildAvailableSkillsBlock` in `skills.service.ts`) emits an `<active_skills><skill name=… source=…>FULL SKILL.md BODY</skill>…</active_skills>` block plus a one-line directive ("Apply these playbooks to the current user request."), which is appended as an **additional trailing `role: "system"` message** to `modelMessages`. **Crucially, the active-skills block is NOT added to the cached `instructions` field** — like the todos block, this is per-turn data that would otherwise invalidate the bit-identical `instructions` cache prefix the moment the user toggled between different `/<skill>` commands across turns. See the prompt-caching paragraph in **Context metering + auto-compaction** for the cache-prefix invariants.
- Synthetic `use_skill` invocation (chat-UI surface): right after the `<active_skills>` injection, the stream layer also inserts one `tool_invocations` row per resolved skill (status `success`, tool name `use_skill`, input `{ name }`, output exactly matching `UseSkillFoundOutput` from `use-skill.ts` — including `files` from `listSkillFiles(skill)`) tied to the assistant placeholder message id, then emits a `tool-call` + `tool-result` SSE pair. The frontend's existing `ToolCallCard.tsx` `use_skill` rendering picks them up unchanged, so a slash-loaded skill looks identical in chat to one the model called itself. The synthetic invocation is allocated a real `messageSeq` (`allocateMessageSeq()`) so it sorts ahead of the assistant's text/tool-calls in the same turn.
- Out of scope for v1: multiple slash commands per message, custom user-defined slash commands (e.g. `~/.agnt/commands/<name>.md`), `/compact` / `/new` / `/clear`, inline skill arguments (`/skill arg=foo`). The `SlashCommand[]` abstraction in the popover already covers a future "user commands" source dropping in alongside skills.

### Agent tools
- Tool *definitions* live in `server/src/modules/conversations/tools/` as plain `{ name, description, inputSchema, execute }` objects (`ToolDefinition`). Registry: `server/src/modules/conversations/tools/index.ts` exports `AGNT_TOOL_DEFS` plus an `UNGATED_TOOL_NAMES` set of tools that bypass the permission gate entirely.
- `conversation.stream.ts` builds a per-conversation tool set via `buildConversationTools({ conversationId, mode })` (`server/src/modules/conversations/permissions/with-permission.ts`) which wraps each definition's `execute` through `withPermission`, EXCEPT for tools listed in `UNGATED_TOOL_NAMES` (currently `question`), whose `execute` is passed through unwrapped. The tools are passed to `streamText` with `stopWhen: stepCountIs(Infinity)`.
- Permission gate: each tool call is decided by (1) conversation permission mode (`ask`/`bypass`, effective-state key `permissionMode`), (2) the per-tool setting from the `toolPermissions` settings category (`ask`/`allow`/`deny`), and (3) the in-memory session-allow cache (`allow_session` decisions). Allow-by-default built-ins are explicitly listed in code (`read_file`, `glob`, `grep`, `use_skill`); unlisted future tools default to `ask`. `deny` short-circuits with an error. `ask` routes through `requestPermission` which returns a promise resolved by the frontend. Ungated tools skip this whole flow.
- Tool invocations are persisted in `tool_invocations`: a row is inserted on `tool-call` (status `pending`) and updated on `tool-result` (`success`) / `tool-error` (`error`). Pending rows are marked `error` when the stream aborts or errors.
- SSE protocol events: `tool-call`, `tool-result`, plus permission events `permission-required` (`{ id, messageId, toolName, input, createdAt }`) and `permission-resolved` (`{ id, messageId, decision }`), plus question events `questions-required` (`{ id, messageId, questions, createdAt }`) and `questions-resolved` (`{ id, messageId, answers }`), plus `plan-updated` (`{ conversation_id, plan: { id, title, content, todos, filePath, createdAt, updatedAt } }`), plus `assistant-model` (`{ messageId, modelId }`) emitted right after model resolution so the assistant-message footer can render the model label while the stream is still running. Each carries `messageId` so the frontend attaches them to the right assistant message. `Message.tool_invocations` is part of the conversation fetch payload.
- Permission HTTP endpoint: `POST /workspaces/:id/conversations/:conversationId/permissions/:requestId/respond` with `{ decision: "allow_once" | "allow_session" | "deny" }`. Deleting a conversation also calls `clearConversationPermissionState` to reject pending requests and clear the session-allow cache.
- Question HTTP endpoint: `POST /workspaces/:id/conversations/:conversationId/questions/:requestId/respond` with `{ answers: string[][] }` (one inner array per question, each containing 1+ selected labels or custom-typed strings). Server validates shape, cardinality, and single-select constraints. Deleting a conversation also calls `clearConversationQuestionState` to reject pending requests.
- Tool listing endpoint: `GET /tools` returns `[{ name, description }]` used by the settings panel. Ungated tools (e.g. `question`) are filtered out of this listing so they never show in `Tool permissions`.
- Current tool set:
  - `read_file(path, maxBytes?)` — reads a utf-8 file. Accepts absolute paths, workspace-root-relative paths (leading `/` or `\`), or paths relative to the workspace. Rejects binary (NUL-byte scan), default 256KB cap, hard cap 1MB. Located at `server/src/modules/conversations/tools/read-file.ts`.
  - `glob(pattern, path?, limit?)` — finds files by glob pattern inside the workspace. Path rules follow `read_file`, but absolute paths are only accepted if they live inside the workspace (containment enforced via `resolveWorkspacePath`). Default ignore list prunes `node_modules`, `.git`, `dist`, `build`, `.next`, `target`, `out`, `.venv`, `venv`, `__pycache__`, `coverage`, editor caches, etc. Default limit 100, hard cap 500, hard cap 50k dir entries scanned. Located at `server/src/modules/conversations/tools/glob.ts`.
  - `grep(pattern, path?, include?, caseInsensitive?, maxResults?)` — regex search over workspace files. Shares path/containment rules and ignore list with `glob`. Skips symlinks, binary files (NUL-byte scan), empty files, and files >1MB; global scan cap 50MB total and 50k dir entries. Default 100 matches, hard cap 1000; lines truncated at 400 chars. Located at `server/src/modules/conversations/tools/grep.ts`.
  - `use_skill(name)` — loads a skill playbook by name from the discovery roots listed under **Skills** above. Skill list for the current stream is injected via `createUseSkillToolDef(() => skills)` in `buildConversationTools`. Located at `server/src/modules/conversations/tools/use-skill.ts`.
  - `image_gen({ prompt })` — generates one PNG via the ChatGPT Codex built-in `image_generation` tool on `https://chatgpt.com/backend-api/codex/responses`. Requires an active Codex OAuth connection (same auth as the conversation stream); API-key-only setups are not supported. Model is resolved from the conversation's active model when it has `supportsImageInput: true`, otherwise falls back to `gpt-5.4`; a single-retry fallback to `gpt-5.4` also kicks in on backend error. The returned base64 PNG is persisted through `createAttachment` + `linkAttachmentsToMessage` tied to the current assistant turn, and the tool output `{ ok, attachmentId, fileName, mimeType, prompt, revisedPrompt, model }` is rendered inline inside `ImageGenBlock` in `ToolCallCard` via the existing `/workspaces/:id/attachments/:id/content` endpoint. Defaults to `ask` permission (unlisted tools default to `ask`) so the user approves before credits are spent. Located at `server/src/modules/conversations/tools/image-gen.ts`; requires `getAssistantMessageId` on the permission context (plumbed from `conversation.stream.ts`).
  - `write_plan({ title, content, todos: [{ id?, content }] })` — ungated plan-mode tool. Creates or updates the conversation's implementation plan. Writes markdown to `~/.agnt/plans/plan-<uuid>.md`, upserts into the `plans` table, and emits `plan-updated` SSE. `toModelOutput` returns a compact text confirmation. Only available in plan mode (filtered out in agent mode). Located at `server/src/modules/conversations/tools/write-plan.ts`.
  - `todo_write({ todos: [{ id?, content, status }] })` — ungated planning tool. Atomically REPLACES the conversation's todo list (omitted ids are deleted, new ids minted server-side, existing ids preserve `created_at`). At most one item may be `in_progress` per call (server rejects otherwise). Returns `{ ok: true, todos, counts }`. Located at `server/src/modules/conversations/tools/todo-write.ts`. Persists into the `todos` SQLite table via `server/src/modules/conversations/todos/todos.service.ts`, which also publishes a per-conversation pub/sub the stream subscribes to in order to emit `todos-updated` SSE. The current todos are re-injected into the system prompt every turn via `buildTodosPromptBlock` (consumed in `conversation.prompt.ts`), so the model always sees its own plan without a separate read tool. Counted toward `/context` breakdown under `todos`.
  - `write({ path, contents })` — create a new file or overwrite an existing one with the full UTF-8 `contents`. Path rules match `glob`/`grep` (workspace-containment enforced via `resolveWorkspacePath`; absolute paths must live inside the workspace). Missing parent directories are created automatically and reported back in `createdDirectories`. When overwriting a file that already uses CRLF line endings, incoming LF-only content is transparently normalized to CRLF so edits don't silently flip line-ending style. Hard-capped at 10 MiB per call; refuses to write when the target path exists but isn't a regular file. Description instructs the agent to PREFER `str_replace` over `write` for edits to existing files. Defaults to `ask` permission. Located at `server/src/modules/conversations/tools/write.ts`.
  - `str_replace({ path, old_string, new_string, replace_all? })` — exact string replacement inside an existing text file. Path rules match `write`; file must already exist (use `write` to create). `old_string` must match verbatim including whitespace/indentation; when `replace_all` is false (default) the tool fails unless `old_string` occurs exactly once. `replace_all: true` replaces every occurrence (use for file-wide renames). Identical `old_string`/`new_string` is rejected as a no-op. CRLF compatibility: if the file is CRLF but the incoming `old_string` uses LF only, both `old_string` and `new_string` are normalized to CRLF before the search/replace so the model can quote text it received through `read_file` (which normalizes to LF). Binary files are rejected via NUL-byte scan; file size capped at 10 MiB. Defaults to `ask` permission. Located at `server/src/modules/conversations/tools/str-replace.ts`.
  - `apply_patch({ input })` — apply an OpenAI V4A diff envelope (`*** Begin Patch` … `*** End Patch`) that can `*** Add File`, `*** Update File` (optionally followed by `*** Move to: <newPath>`), and/or `*** Delete File`, with `@@ <anchor>` lines and ` `/`-`/`+` hunk-line prefixes. This is the preferred edit tool on Codex/OpenAI models (that format is in their post-training distribution); `write` and `str_replace` remain available for single-file edits. One call can mutate many files at once. Parser accepts a leading ```/```diff fence or an `apply_patch <<EOF` heredoc wrapper. Server parses the whole envelope, resolves every path through `resolveWorkspacePath` (workspace-containment enforced), pre-flight-checks existence constraints (`Add File` fails if target exists; `Update File` / `Delete File` fails if missing), then applies in memory before writing anything — so a malformed hunk aborts the tool without leaving a half-patched tree. Hunk matching uses `@@ <anchor>` lines to narrow the search window (anchors do substring-match on a line; stack multiple for nested scopes) and requires the combined ` `/`-` before-text to occur exactly once within that window; descriptive errors distinguish "anchor not found", "context not found", "context matches N places". Line endings are matched flexibly (LF/CRLF) and the original EOL style is restored per file. `Move to:` writes the updated content to the new path and `rename`s away from the original (rejects if the destination already exists). Missing parent directories for `Add File` / `Move to:` are auto-created and reported in `createdDirectories`. Hard caps: 10 MiB per file, 32 MiB for the whole patch text. Returns `{ ok: true, changes: [{ op: "add"|"update"|"delete"|"rename", path, relativePath, newPath?, newRelativePath?, oldContents, newContents, linesAdded, linesRemoved, createdDirectories }], summary: { filesChanged, filesAdded, filesDeleted, filesUpdated, filesRenamed, linesAdded, linesRemoved } }` — full old/new contents are included for the frontend diff view. `toModelOutput` narrows what the model receives to a compact textual summary (`Applied N files (+a -b):` + one line per change) so per-file contents don't re-enter the context. Defaults to `ask` permission. Located at `server/src/modules/conversations/tools/apply-patch.ts`.
  - `question({ questions: [{ question, header (≤30 chars), options: [{ label, description }], multiple }] })` — ungated UI tool. Presents one or more multiple-choice questions in the chat input (replacing it, same UX pattern as the permission card) and blocks until the user answers. A "Type your own answer" pill is always rendered client-side, so the model must NOT include catch-all options like "Other"; if an option is recommended, put it first and append ` (Recommended)` to its label. Returns `{ answers: string[][] }` — each inner array is the selected option labels (or the user-typed custom string) for the matching question; even single-select answers come back as a 1-element array. Located at `server/src/modules/conversations/tools/question.ts`; blocks on `requestQuestions` from `server/src/modules/conversations/questions/gate.ts`. This tool bypasses the permission gate entirely and is hidden from the `Tool permissions` settings panel.
  - Workspace-containment + ignored-dir logic lives in `server/src/modules/conversations/tools/workspace-path.ts` and is shared across filesystem-walking tools.

Pending permission requests are surfaced by the `PermissionCard` above the chat textarea and the sidebar swaps `MinusIcon` for a pulsing `ShieldWarningIcon` on the affected conversation. Pending question requests are surfaced by `QuestionCard` (`app/src/components/chat/QuestionCard.tsx`) which takes priority over `PermissionCard` when both are active and renders all questions in one stacked scrollable card with selectable option pills, a dashed "Type your own" pill that reveals an input, and a single Submit button gated on every question having at least one answer. ### Context metering + auto-compaction
- Server-authoritative token usage: AI SDK `streamText` `onFinish` persists `input_tokens`/`output_tokens`/`reasoning_tokens`/`total_tokens` to `messages` and emits a `finish` SSE event with `usage` for the just-completed assistant turn.
- `GET /workspaces/:id/conversations/:conversationId/context` returns `{ modelId, contextWindow, maxOutputTokens, usedTokens, percent, breakdown: { messages, reasoning, toolOutputs, attachments, repoInstructions, systemInstructions, todos }, messageCount, compactedMessageCount, hasCompactSummary, lastCompactedAt, autoCompactThreshold }`. `hasCompactSummary` is computed strictly from rows where `role='system' AND summary_of_until IS NOT NULL` (the field is set exclusively by `compactConversation` so it's the authoritative marker — generic system messages never count). Attachment tokens use stored `estimated_tokens` (fallback re-estimation for legacy rows): text tokenized via `gpt-tokenizer` o200k_base, images = 1105 per image, PDFs ≈ bytes/3. `repoInstructions` tracks injected `AGENTS.md` / `CLAUDE.md` prompt content separately from the base system prompt.
- Auto-compaction runs **inside** the SSE stream so the user never sees a UI freeze while the LLM summarizes. The decision/emission/run is shared by `streamConversationReply` (incoming user message) and `streamReplyToLastMessage` (first reply on a freshly created conversation) via the helper `runAutoCompactionForStream(workspaceId, conversationId, controller, extraTokens)`. The helper computes `projected = currentContext.usedTokens + extraTokens` (tokens not yet in the DB — pending user message content + attachment estimates + mention block; pass 0 when the user message is already persisted), and if `projected / contextWindow > COMPACT_THRESHOLD` it emits `compaction-started`, awaits `compactConversation(...)`, and emits exactly one of `compacted` / `compaction-skipped` / `compaction-error`. Sequence in `streamConversationReply`: (1) insert the user row + emit `user-message` SSE so the chat shows the bubble immediately, (2) call `runAutoCompactionForStream(..., 0)` (the user row is now in the DB, so its tokens are already counted by `computeContextSummary`), (3) build the model history AFTER compaction so `streamText` sees the smaller context, (4) insert the assistant placeholder, emit `assistant-start`, run the normal generation loop. Sequence in `streamReplyToLastMessage` (the home-screen-create-then-reply path used by `/reply`): identical, except step 1 is skipped — the user message was already persisted by `createConversation`. Until 2026-04-26 the `/reply` route had **no** compaction logic at all, so a freshly created conversation could overflow the model's context window before any compaction ever ran; that's now fixed by routing through the same helper.
- `compactConversation` summarizes all messages older than the last `COMPACT_KEEP_RECENT = 6` (3 user/assistant pairs) plus always keeps the most recent user message; older rows are marked `compacted=1` and a single `role='system'` message with `summary_of_until=<last summarized id>` holding the summary is inserted. SSE shapes: `compacted { summaryMessageId, summarizedMessageIds, summarizedCount, usedTokensAfter, summaryContent, summaryCreatedAt, summaryOfUntil }`, `compaction-skipped { conversation_id, reason }`, `compaction-error { conversation_id, error }`. On error the stream continues without compaction; the model may itself error if context overflows, surfaced via the existing `error` event.
- **Summarizer input fits the summarizer model.** The summarizer LLM call is itself bound by the active model's `contextWindow`, and a single old message can easily contain a 100k-token tool-output blob. To prevent the summarizer from overflowing its own context, `compactConversation` truncates each message to `COMPACT_MAX_PER_MESSAGE_CHARS = 8000` chars (80% head + 20% tail with an explicit `[... N chars truncated for summarizer ...]` marker), then halves the per-message cap (down to a 512-char floor) until the joined input fits `inputCharBudget = max(16k, modelContextWindow * 4 * COMPACT_INPUT_BUDGET_FRACTION)` (where `COMPACT_INPUT_BUDGET_FRACTION = 0.6` reserves room for the system prompt + reasoning + output). If many rows still overflow, a final hard tail-clip with a head marker keeps the most recent portion. The `[compact] Summarizing` log line emits `modelContextWindow` / `inputCharBudget` / `finalInputChars` / `perMessageCap` / `hardClipped` for forensics.
- **Tool-output rewrite during compaction.** Compacting `messages` rows alone is rarely sufficient — in tool-heavy workflows, almost all the context lives in `tool_invocations.output_json`. The kept-recent 6 messages each typically reference 5–20 tool invocations whose outputs total 25-30k tokens, so even a perfect summary of older messages leaves post-compaction usage at 80%+ of the window, immediately re-tripping the 85% threshold next turn. To break that loop, `compactConversation` rewrites every `tool_invocations.output_json` that either (a) belongs to a message about to be marked `compacted=1` or (b) has `LENGTH(output_json) >= COMPACT_TOOL_OUTPUT_TRIM_AT_CHARS` (4000 chars) and belongs to a kept message — replacing the JSON value with a sentinel object `{ __agnt_compact_trimmed: true, originalChars, toolName, trimmedAt, placeholder }`. The replay path (`toolResultOutputFromInvocation`) detects the sentinel via `isCompactTrimmedOutput(output)` and short-circuits to `{ type: "text", value: placeholder }` BEFORE the per-tool `toModelOutput` runs (the sentinel doesn't match any tool's expected output schema, so `toModelOutput` would otherwise crash). The frontend's `ToolCallCard` re-declares the same sentinel shape + guard locally and falls through to a generic `CompactionTrimmedBlock` rendering ("`<toolName>` · output trimmed (`<N>` chars freed)") instead of the per-tool block. This rewrite is destructive — the original output is not recoverable from the DB — but that's intentional: anything less wouldn't actually free the context. The `[compact] Done` log line emits `toolOutputsTrimmedCount` and `toolOutputCharsSaved` so it's easy to verify how much was reclaimed.
- **Deterministic fallback so compaction never fails.** When the summarizer LLM call fails (provider 4xx/5xx, network drop, prompt still too big despite truncation, empty completion, etc.) `compactConversation` no longer throws — it builds a deterministic structured-markdown summary via `buildDeterministicSummary(rows)` (`## Conversation summary` self-describing as fallback, `## Compacted messages` listing role + timestamp + size + 220-char preview for up to 40 rows, then `## Open questions / todos` + `## Important references`) and uses that as the summary content. The rest of the function still runs: the `role='system'` summary row is inserted with `summary_of_until` set, the older rows are marked `compacted=1`, and `computeContextSummary` reflects the freed-up space. This means the upstream `runAutoCompactionForStream` always emits `compacted` (with the fallback markdown) instead of `compaction-error`, so the frontend's `CompactionMarker` always shows up and `compactingByConversationId` is reliably cleared. Without this, a single failing summarizer call would put the conversation in a permanent overflow loop, since every subsequent turn would re-attempt the same failing summarization. The `[compact] Done` log line includes `summarizerFailed: true|false` so it's easy to tell from logs whether a given compaction used the LLM or the deterministic fallback. Original message bodies are still untouched in the DB (just `compacted=1`), so audit/inspection paths are unaffected.
- **Mid-turn trim.** `compactConversation` runs *between* user turns; it summarizes messages already persisted in the DB. But a single assistant turn can accumulate hundreds of kilobytes of tool output (`read_file`, `grep`, etc.) inside `streamText`'s tool loop, and those live entirely in memory until the turn finishes — without intervention the loop sends the entire growing transcript to the model on every step until we cross the model's hard context limit. `runStreamTextIntoController` therefore wires a `prepareStep` callback (`trimOversizedToolResultsForStep`) into every `streamText` call. Per step it estimates `messages` token count using a coarse `chars / 4` heuristic; if the estimate exceeds `model.contextWindow * COMPACT_THRESHOLD` it walks the messages oldest-first and rewrites every tool-result message whose `output.value` is `>= TOOL_RESULT_TRIM_AT_CHARS` (4000 chars) with a short text placeholder (`[Output trimmed mid-turn to fit context window. Original length: N chars. Re-run the tool if you still need this content.]`). The trim preserves `toolCallId` and `toolName` so the SDK still pairs each tool-call with its result — only the result body shrinks. A closure-scoped `Set<string>` of trimmed `toolCallId`s makes the trim sticky: subsequent steps re-apply the same trims (the SDK rebuilds `stepInputMessages` from its untrimmed internal record on each step). The first trim of a turn emits a single `mid-turn-trim { conversation_id, message_id, trimmed_count }` SSE event (subsequent steps trim silently); the frontend currently just `console.info`s it for debugging. This intervention only touches the in-flight prompt sent to the model — the original tool output stays in `tool_invocations` rows verbatim, so the chat UI and any subsequent `compactConversation` call still see the full content.
- Prompt caching: the Responses API `instructions` field is kept byte-identical across turns for the same conversation (base instructions + workspace block + repo instructions + warnings + skills block). `conversation.stream.ts` sends `promptCacheKey = conversationId` so every turn routes to the same cache node (mirrors `codex-rs/core/src/client.rs`). `promptCacheRetention` is intentionally NOT sent — the ChatGPT backend (`chatgpt.com/backend-api/codex/responses`) rejects it with 400 "Unsupported parameter: prompt_cache_retention"; that knob only exists on the direct OpenAI Platform API, and the Codex CLI doesn't set it either on the ChatGPT-auth path. The `<Current Todos>` block is NOT part of `instructions`; it is injected as a trailing `role: "system"` message on each turn so todo edits never invalidate the cached prefix. The same trailing-system-message pattern is also used for the per-turn `<active_skills>` block emitted when the user picks a `/<skill-name>` slash command (see **Slash commands**); it carries the full `SKILL.md` body for that turn only, so toggling between different `/<skill>` commands across turns never invalidates the `instructions` cache prefix either. `reasoningSummary: "detailed"` is only sent when `reasoningEffort` is active.
- `POST /workspaces/:id/conversations/:conversationId/compact` triggers the same pipeline manually (from the meter popover at ≥85%).
- History queries for model prompting filter `compacted=0`; the system summary row is included verbatim.
- Frontend rendering: `MessageList.tsx` collapses any run of `compacted=1` rows into a single `CompactionMarker` rendered at the position of the next `role='system' AND summary_of_until` row. The marker is a `<details>` element with the headline `Chat context summarized · N older messages` and a collapsible body that renders the full summary markdown via `MarkdownRenderer` (the legacy `[Compacted summary of N earlier messages]` prelude inside the stored content is stripped client-side via `stripCompactionPrelude` since the visual headline already conveys that info). While the SSE stream is between `compaction-started` and one of `compacted`/`compaction-skipped`/`compaction-error`, the same `MessageList` also renders a transient `CompactionInProgress` indicator (pulsing `ArchiveIcon` + `Summarizing chat context…`) at the bottom of the message list, gated on `compactingByConversationId[conversationId]` from the conversation store.
- Frontend store: `useConversationStore` has a `compactingByConversationId: Record<string, true>` slice and a `setCompacting(conversationId, boolean)` action. The SSE reducer in `handleConversationSseEvent` flips it to `true` on `compaction-started` and back to `false` on `compacted` / `compaction-skipped` / `compaction-error`. As a safety net it is also force-cleared on the terminal stream events `finish` / `abort` / `error`, so a dropped/crashed SSE never leaves the indicator stuck on. `compaction-error` additionally `console.warn`s the error message — it is non-fatal because the stream continues without compaction.
- Frontend: `app/src/features/context/` provides `useContextMeter` (merges server summary with client-side tokenization of the draft via `gpt-tokenizer` and pending attachment token estimates). `ContextMeter.tsx` renders an 18px SVG ring with `strokeLinecap="butt"`, color tiers, tooltip breakdown, and a popover at ≥85% that calls `/compact`. Mounted immediately left of the send/stop button in `ChatInput`.
- Deps: `gpt-tokenizer` added to both `server/` and `app/`.

### Global stats dashboard
- `GET /stats?tzOffsetMinutes=<int>` returns lifetime stats from an append-only ledger at `~/.agnt/stats.db`. Payload shape: `{ totals: { sessions, userMessages, inputTokens, outputTokens, reasoningTokens, totalTokens, activeDays }, streak: { current, longest }, favoriteModel: { id, label, count } | null, models: { id, label, count }[] (sorted desc), hours: number[24], mostActiveHour: number | null, heatmap: { startDate, endDate, days: { date, count }[210] }, workspaceCount }`.
- `tzOffsetMinutes` is minutes east of UTC (`-new Date().getTimezoneOffset()`), range `−840..840`, default `0`. Applied as a SQLite datetime modifier (`date(created_at, '${sign}${abs} minutes')`) so day/hour buckets line up with the user's local timezone.
- Stats DB is completely separate from the per-workspace conversation DBs. It is NEVER touched by conversation deletion, so deleting a chat does not reduce any lifetime stat. Rows are insert-only and live at `~/.agnt/stats.db`.
  - Module: `server/src/lib/stats-db.ts` bootstraps the DB (WAL + foreign keys) with two tables:
    - `stat_sessions (id, workspace_id, conversation_id, created_at)` — one row per top-level conversation ever started (subagent children are skipped; they're internal plumbing, not user-facing sessions).
    - `stat_messages (id, workspace_id, conversation_id, message_id, role, model_id, input_tokens, output_tokens, reasoning_tokens, total_tokens, created_at)` — one row per user message (tokens NULL) or per assistant turn (tokens + model populated at `onFinish`). `role` is constrained to `'user' | 'assistant'`; system/compaction summaries are not recorded.
    - Indexes: `idx_stat_sessions_created`, `idx_stat_messages_created`, `idx_stat_messages_role_created`, `idx_stat_messages_model`.
- Recorder at `server/src/modules/stats/stats.recorder.ts` exposes best-effort writers (`recordStatSession`, `recordStatUserMessage`, `recordStatAssistantMessage`) that swallow errors so stats never break a conversation flow.
- Hook points:
  - `createConversation` (`conversations.service.ts`) records a session AND the first user message into the ledger.
  - `addMessage` route records user rows only (assistant turns come from the stream onFinish below).
  - `streamConversationReply` (`conversation.stream.ts`) records the appended user message immediately after the `INSERT INTO messages` that persists it.
  - `runStreamTextIntoController` records an assistant row inside `streamText.onFinish` using the resolved `modelName` + final `usage` + the `assistantCreatedAt` timestamp threaded in from every call site (primary reply, continue-reply, subagent stream). Aborted turns with no finish do not emit a row.
  - `createSubagentConversation` does NOT record — subagent activity is excluded by design.
- Service at `server/src/modules/stats/stats.service.ts` now queries only the stats DB. Streaks still computed in JS from `Set<YYYY-MM-DD>` of user-message local days, relative to the user's local "today". Sessions count = `COUNT(*) FROM stat_sessions`. Heatmap window is a rolling 210 days (30 columns × 7 rows). `workspaceCount` still comes from `listWorkspaces()`.
- The `messages.model_id` column in each workspace DB is still populated during the stream (existing behavior preserved) but is no longer read by the stats service — it remains for any per-conversation UI that may want it.
- Frontend (unchanged shape): `app/src/features/stats/` (types + API client + `useGlobalStats` hook) and `app/src/components/stats/` (`StatsPanel`, `StatCard`, `UsageHeatmap`). The panel is a single compact card rendered on the `/` route above the chat input with one segmented-tab row: `Overview`/`Models` (content toggle). Overview shows a 4×2 tile grid: Sessions · Messages · Total tokens · Active days / Current streak · Longest streak · Peak hour · Favorite model. Models view replaces the tiles with a ranked bar-chart list by model usage. The heatmap always renders the full 210-day window below the tiles with a blue 5-tier scale on `dark-800` cells. A playful footer compares total-token usage to familiar reference texts (Gatsby, Harry Potter, Bible, etc.). Counts user messages only.

### Tauri sidecar lifecycle
- `app/src-tauri/src/lib.rs` owns the sidecar child handle (`SidecarState { child: Arc<Mutex<Option<CommandChild>>> }`) and only spawns the sidecar in **release builds** (`#[cfg(not(debug_assertions))]`). Spawn happens during `Builder::setup` with fixed args `serve --port 4727 --hostname 127.0.0.1`. The frontend, also in release mode, resolves `SERVER_BASE_URL` to `http://127.0.0.1:4727` via `app/src/lib/server-url.ts` so both sides agree. No password is generated — `SERVER_PASSWORD` is only enforced if explicitly set in `server/.env`. `stdout`/`stderr` from the sidecar are forwarded to the host process via `[SIDECAR]` / `[SIDECAR ERR]` log prefixes through `tauri::async_runtime::spawn` reading the `CommandEvent` stream.
- Spawn errors (e.g. port 4727 already in use) are logged but do NOT crash the app — the frontend's existing health monitor (`probeHealth` polling every 3s) will still find whatever else is listening on 4727.
- In **debug builds** (`tauri dev` — both `bun run prod` and `bun run local:dev`) `spawn_sidecar` is `cfg`-compiled out entirely; no sidecar is started by Rust. The developer keeps `bun run start:server` running in a separate terminal on **port 4728** so server-side changes hot-reload via `bun --watch`. The dev frontend bundle resolves `SERVER_BASE_URL` to `http://127.0.0.1:4728` via the same `import.meta.env.DEV` switch in `server-url.ts`. The two-port split (4727 prod / 4728 dev) is intentional so a developer can run the built/installed app and a `bun run start:server` simultaneously without clashing.
- On window close (`WindowEvent::CloseRequested`) the sidecar child process is killed via `kill_existing_server` (`child.kill()` on the stored handle) and every interactive terminal PTY is killed via `terminals::kill_all` (see **Terminals (sidebar)**).
- There is no Tauri command to start/stop the sidecar from the frontend — the lifecycle is owned entirely by Rust and tied to the app process.

### Terminals (sidebar)
- Real interactive PTYs live in Rust via `portable-pty` (`portable-pty = "0.9"` in `app/src-tauri/Cargo.toml`). The user-facing terminal stack is **separate from** the agent's `shell` tool runtime in `server/src/modules/conversations/shell/*` — they share no code; the server-side runtime is still a non-PTY `child_process.spawn` used only for the AI agent's `shell` tool.
- Module: `app/src-tauri/src/terminals.rs` exposes a `TerminalsState { sessions: Arc<Mutex<HashMap<String, Arc<TerminalSession>>>> }` registered via `app.manage(TerminalsState::default())` in `lib.rs`. Each `TerminalSession` owns the master pty (`Mutex<Box<dyn MasterPty + Send>>`), the writer (`Mutex<Box<dyn Write + Send>>`), and a clone of the child killer (`Mutex<Box<dyn ChildKiller + Send + Sync>>`).
- Tauri commands (registered in the `invoke_handler!` macro in `lib.rs`):
  - `terminal_open({ id, workspaceId, cwd, cols, rows, shell? })` — `native_pty_system().openpty(...)`, spawns a `CommandBuilder` with the picked shell (custom override > `powershell.exe` on Windows > `$SHELL` else `/bin/bash` on POSIX), saves the session, and spawns two threads: a reader thread looping on `master.try_clone_reader()` that emits `terminal://output { id, data }` (8 KiB read buffer, UTF-8 lossy decode) and a wait thread that calls `child.wait()`, removes the session from the map, then emits `terminal://exit { id, code }`.
  - `terminal_write({ id, data })` — write bytes to the saved writer + flush.
  - `terminal_resize({ id, cols, rows })` — `master.resize(PtySize { rows, cols, ... })`.
  - `terminal_close({ id })` — pop the session, kill the child via the cloned killer.
  - `terminal_list_alive()` — returns the keys of the live session map. Used on app boot by the frontend to detect if a persisted descriptor's PTY survived a webview reload (so we re-attach instead of respawning).
- Window-close cleanup: the existing `CloseRequested` handler in `lib.rs` now also calls `terminals::kill_all(&state)` alongside the sidecar kill, so no stray PTYs survive app exit.
- Frontend module: `app/src/features/right-sidebar/terminals/`:
  - `terminal-types.ts` — `TerminalDescriptor { id, workspaceId, name, cwd, createdAt }` + event payload types.
  - `terminal-bridge.ts` — thin wrappers over `@tauri-apps/api/core` `invoke` and `@tauri-apps/api/event` `listen`; opens **one** global listener per channel (`terminal://output` and `terminal://exit`) and fans out to per-id callbacks via `subscribeOutput(id, cb)` / `subscribeExit(id, cb)`.
  - `terminal-store.ts` — Zustand `persist`'d store with `terminalsByWorkspace: Record<string, TerminalDescriptor[]>`, `activeIdByWorkspace: Record<string, string | null>`, `scrollbackByTerminalId: Record<string, string>` (sliding tail bounded to `SCROLLBACK_LIMIT = 256 * 1024` bytes per terminal), and `sidebarOpen`. Persisted under localStorage key `right-sidebar-terminals`. All mutating actions (`addTerminal`, `closeTerminal`, `setActive`, `appendScrollback`, `renameTerminal`, `toggleSidebar`/`setSidebarOpen`) are workspace-scoped.
  - `terminal-session.ts` — non-React `Map<id, SessionRuntime>` of live xterm.js instances. Each `SessionRuntime` owns a `Terminal` + `FitAddon` + `WebLinksAddon` + a wrapper `<div>` that's the actual xterm host. Wrapper is initially appended to a hidden pool `<div>` (`getPool()` — `position: fixed; left: -99999px; visibility: hidden`) so `term.open()` can be called once on a real DOM node; `mountSession(id, container)` `appendChild`s the wrapper into the visible container, `unmountSession(id)` moves it back to the pool. xterm is **never disposed** on tab/workspace switches — only on explicit `disposeSession(id)` (called from the close-X button) — so scroll position, selection, and exact buffer state survive switches.
  - `terminal-view.tsx` — owns a single `<div ref>`, calls `ensureSession` + `mountSession` on mount and `unmountSession` on cleanup, plus a `ResizeObserver` that calls `fitSession` on container resize. After the (async) `initPromise` resolves it refits once so the freshly opened PTY's size matches the actual visible container.
  - `terminal-sidebar.tsx` — workspace-scoped list of `TerminalItem` rows + `+` button. Close-X calls the store's `closeTerminal` followed by `disposeSession`.
  - `terminals-tab.tsx` (in `app/src/features/right-sidebar/tabs/`) — reads the active workspace from `useWorkspaceStore`, derives the workspace's terminal list + active id from the store (falling back to last entry if persisted active id no longer exists), auto-creates one terminal in `workspace.path` when the active workspace has zero terminals, and proactively `ensureSession`s every terminal in the active workspace's list (so inactive-but-existing terminals also have live PTYs in the background).
- Lifecycle semantics:
  - **Across workspace switches**: PTYs stay alive in Rust (the session `HashMap` is global, not workspace-scoped), and xterm wrappers stay alive in memory but get parked in the hidden pool. Switching back is a `mountSession` move-back into the visible container — instant, with full prior buffer.
  - **Across full app restart**: Rust state is gone. Persisted descriptors + scrollback rehydrate from localStorage. On first `ensureSession` for a given id, the session manager calls `listAliveTerminals()` once, caches the result, and: if the id is in that set → re-attach (don't reopen, just resubscribe); if not → write the persisted scrollback verbatim, append a dim `[Restored from previous session — starting a fresh shell]` banner, then call `terminal_open` with the saved `cwd` and the current xterm `cols`/`rows`. The replayed scrollback above the live prompt is read-only history (it's already inside the xterm scrollback buffer).
  - **Explicit close (X)**: kills the PTY in Rust (`terminal_close`), drops the descriptor + the persisted scrollback for that id, disposes the xterm.
- xterm.js setup: `@xterm/xterm@6` + `@xterm/addon-fit` + `@xterm/addon-web-links`, dark theme matching `--color-dark-*` tokens, 12px font, 5000-line scrollback, `cursorStyle: "bar"`, `convertEol: false` (PTY already produces CRLF). CSS imported via `@xterm/xterm/css/xterm.css` from `terminal-session.ts`.

### Chat input draft persistence
- Feature module: `app/src/features/chat-drafts/` — `chat-drafts-store.ts` is a Zustand `persist` store keyed on `localStorage` under the name `chat-input-drafts` (mirroring the shape of `right-sidebar/terminals/terminal-store.ts`). State shape: `{ homeByWorkspace: Record<workspaceId, DraftSnapshot>, byConversation: Record<conversationId, DraftSnapshot> }`. `DraftSnapshot = { docJSON, plainText, updatedAt }` — `docJSON` is the raw Tiptap `JSONContent` so `@mentions` rehydrate as proper mention nodes (re-parsing `@…` plain text would be fragile). `partialize` only persists those two maps; everything else stays in-memory.
- Slot model: a `DraftSlot` is a tagged union `{ kind: "home", workspaceId } | { kind: "conversation", conversationId }`. `ChatInput` derives its slot from props (`conversationId` wins; otherwise `workspaceId`; otherwise no-slot ephemeral mode). `setDraft` drops effectively-empty drafts (no plain text + no mention nodes) instead of writing `""`, and skips persistence when `JSON.stringify(docJSON).length` exceeds 64 KiB (logs a warning) to avoid pathological pastes blowing out localStorage.
- `MentionEditor` exposes `getDocJSON()` / `setDocJSON(json)` on its imperative handle in addition to `clear` / `focus` / `serialize` / `isEmpty`. `setDocJSON` calls `editor.commands.setContent(json, { emitUpdate: false })` so rehydrating doesn't immediately echo back into the store. The component also accepts an `initialContent?: unknown` prop that's threaded into `useEditor`'s `content` (captured once via `useRef`) — `ChatInput` passes the latest `getDraft(slot)?.docJSON` so when the editor remounts mid-session (e.g. a `PermissionCard`/`QuestionCard` conditionally replaces the form, then dismisses) it picks the persisted draft back up.
- `ChatInput` keeps a `pendingSnapshotRef` containing the last keystroke's `{ slot, snapshot }` and a 250 ms debounced `setTimeout`; the snapshot is captured synchronously in `handleEditorChange` (not pulled from `editorRef.current` later) so the editor unmounting mid-debounce can't drop the user's last few keystrokes. Slot transitions and component unmount both call `flushSaveTimer` which writes the pending snapshot synchronously before clearing the timer. `handleSend` clears the editor, cancels the debounce, drops the pending snapshot, and calls `clearDraft(slot)` so a sent message never leaves a ghost draft.
- Cleanup: `deleteConversation` in `app/src/features/conversations/conversation-store.ts` calls `clearDraft({ kind: "conversation", conversationId })` on hard delete (alongside the existing `clearPending` / `clearTodos` calls). Archive is intentionally NOT cleaned up — archive is reversible and the draft should come back when the conversation un-archives. Pending file attachments are deliberately not part of the draft (transient server uploads); only text + mention nodes survive navigation/restart.

### LSP diagnostics
- Module layout: `server/src/modules/lsp/` — `lsp.types.ts` (wire-compatible `Diagnostic`/`Range`/`DiagnosticsResult`/severity mapping + `isTypeScriptPath`), `lsp.client.ts` (minimal JSON-RPC 2.0 stdio client with `Content-Length` framing, request/notification send, per-URI diagnostic cache, `waitForDiagnostics(uri, { timeoutMs, signal })` that soft-resolves to whatever the server has published so far on timeout, graceful `shutdown()` with SIGTERM/SIGKILL escalation), `lsp.typescript.ts` (`TypeScriptLspProvider` that spawns `typescript-language-server --stdio` via `bun <resolved cli.mjs> --stdio`, tracks per-document `{ version, lastText }`, sends `initialize` with workspace root + minimal client capabilities on first use), `lsp.registry.ts` (per-workspace `Map<workspacePath, { provider, startedAt }>` with lazy `getTypeScriptProvider` + `disposeWorkspace` + `disposeAll`), and `lsp.service.ts` (public `checkFiles` / `checkWorkspace` / `disposeWorkspace` / `disposeAll` facade; centralizes settings lookup + never throws so a dead LSP can't break an edit tool). Binary resolved from `server/node_modules/typescript-language-server/lib/cli.mjs` via `require.resolve` (Bun-compatible); `typescript` + `typescript-language-server` added as `server/` dependencies so the sidecar bundles them.
- Provider lifecycle: lazy-spawn on first call, kept alive across requests so subsequent checks pay only the `textDocument/didChange` + `publishDiagnostics` round-trip. Providers survive idle periods (no timer eviction in v1) and are torn down only on workspace close (`disposeWorkspace`) or server shutdown (`disposeAll`, hooked into the SIGINT/SIGTERM handlers in `server/src/index.ts` with a 2.5s race so graceful close never stalls exit).
- Edit-tool auto-wiring: `write` / `str_replace` / `apply_patch` each call `runPostEditDiagnostics(workspacePath, changedTsPaths)` from `server/src/modules/conversations/tools/post-edit-diagnostics.ts` after a successful mutation. The helper reads the `diagnostics` settings category, filters the inputs to `.ts/.tsx/.js/.jsx/.mjs/.cjs/.mts/.cts` via `isTypeScriptPath`, and returns a `DiagnosticsResult` (or `undefined` when disabled / non-TS / LSP failed). Each tool's `Output` gains an optional `diagnostics?: DiagnosticsResult` field; each tool's `toModelOutput` appends a compact LSP summary (`"LSP: clean."` when healthy, up to 20 `path + line:col + severity + code + message` lines otherwise — truncated message directs the model to call the `diagnostics` tool to see the rest). `apply_patch` skips deleted files when collecting changed paths; rename ops use the new path.
- `diagnostics` agent tool (`server/src/modules/conversations/tools/diagnostics.ts`): `{ path?, scope?: "file" | "workspace", minSeverity? }`. Scope defaults to `"file"` when `path` is set, otherwise `"workspace"`. `"file"` calls `lspService.checkFiles([absolute])`; `"workspace"` calls `lspService.checkWorkspace` which returns diagnostics for every TS/JS document the LSP is currently tracking (v1 doesn't walk the whole filesystem). Workspace-containment enforced via `resolveWorkspacePath`. Output: `{ ok: true, scope, enabled, results: DiagnosticsForFile[], summary: { errors, warnings, infos, hints, filesChecked } }`; `toModelOutput` narrows to a multi-line text block. Registered in `AGNT_TOOL_DEFS`, added to `ALLOW_BY_DEFAULT_TOOL_NAMES` (both backend + frontend mirror), listed in `PLAN_MODE_TOOLS` (read-only), and added to every subagent type's allowlist.
- Settings category `diagnostics` (`enabled` / `autoRunOnEdits` / `minSeverity: "error"|"warning"|"info"|"hint"` / `waitMs: 200..10000`) lives in `app/src/typings/settings.ts` + `server/src/modules/settings/settings.types.ts` (+ `settings.service.ts` normalizer). UI: `app/src/components/settings/DiagnosticsSettings.tsx`, registered in `SettingsPanel.tsx` under the `Desktop` group with a `BugIcon`. Disabling `enabled` short-circuits both auto-run and the `diagnostics` tool (the tool returns an empty result with `enabled: false` so the model knows why it got nothing). `autoRunOnEdits: false` keeps the tool callable while suppressing the post-edit hook.
- Frontend rendering: new `app/src/components/chat/DiagnosticsInline.tsx` exports `parseDiagnosticsResult(unknown)` + the `<DiagnosticsInline />` component (grouped-by-file rows with severity-count pills `3E 1W`, click to expand into `line:col [severity] [source(code)] message` ordered list, color-coded by severity). `ToolCallCard.tsx` renders `<PostEditDiagnostics output={output} />` under the diff in `WriteBlock`/`StrReplaceBlock` and under the change list in `ApplyPatchBlock`, plus a dedicated `DiagnosticsBlock` (BugIcon) for `diagnostics` tool calls themselves.

## Server access pattern (two ports, dev vs prod)
The frontend, server, and Rust host all agree on a single base URL — but the port differs by build mode so dev and prod can run side by side:

- **Production / built app** — `http://127.0.0.1:4727`. Rust auto-spawns the sidecar there (see **Tauri sidecar lifecycle**), and the release frontend bundle resolves `SERVER_BASE_URL` to that URL via `app/src/lib/server-url.ts` (`import.meta.env.DEV === false`).
- **Local development** — `http://127.0.0.1:4728`. The developer runs `bun run start:server` from `server/` (script hard-codes 4728), and the Vite-served frontend resolves `SERVER_BASE_URL` to that URL (`import.meta.env.DEV === true`).

`VITE_API_URL` still overrides everything if explicitly set (e.g. pointing at a remote/staging server).

Auth via `Authorization: Basic app:<password>` is only enforced when `SERVER_PASSWORD` is set in `server/.env`; the sidecar is launched without it by default. If you re-introduce auth, set `SERVER_PASSWORD` in `server/.env` (the build script bakes it into the compiled sidecar) AND set `VITE_API_PASSWORD` for the frontend so both sides match.
